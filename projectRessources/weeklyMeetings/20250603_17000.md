## Gesprächsinhalt:
- zu dem Editor Probiertest:
    - gängige MarkDown Konvention auschecken und die gängige Konvention benutzen (Vgl. GitHub)
- zu dem Korrekturmodus:
    - je nach Zeit und Arbeitsaufwand: .csv Export, sodass die LuL in Excel die Noten ausrechen lassen können

### Client:
- Anmerkung: wenn Token ausgeteilt wird, dann fraglich, ob die Unterschrift Sinn macht
- wie werden die Aufgaben von dem Server an den CLient geschickt => **Konsens:** String mit Markdown; ggf. Verschlüsselung auf die Verbindung legen (SSH)
- Alternativ und erweitert: json, oder andere Datenformate bereits aufbereitet übersenden
- Fortschrittsbalken: ändern zu Frage X von Y (Hinweis wegen untschiedlichen Zeitanforderungen von den AUfgaben)

### Client-Chatfunktion:
- schöne Idee. Aber sehr aufwändig => ggf. in der FOrm überdenken
- ggf. ersetzten durch Lehrer rufen

### Client-Abgabe:
- wenn der Abgaben Button gedrückt wird, vor dem Abgeben noch eine kleine Zusammenfassung anzeigen lassen => dann soll gefragt werden, ob sich der S mit den Antworten sicher ist

### Server-Token:
- analoge Liste nicht, sondern digital tokens den SuS zuweisen => dementsprechend entfällt der Punkt mit der DSGV=

### Server-Testat-Verwaltung:
- das Importieren hinten heranstellen
- erweiterte Idee: ggf. die Testate speichern auf dropbox, oder google drive => in punkto Datenschutz können die Schulen die Schulclouds einbinden (dazu muss der anwendende Leherer die Kompatibilität testen)
- Anregung: wenn Testate imoportiert werden, dann können die Attribute klasse und Schuljahre automatisch ausgelesen werden und dann in entsprechende Unterordner abgelegt werden
- **in Dokumentation die Einschränkung bzw Ordnerkonvention für die TEstatdaten vermerken => insbesondere um auf mögliche Probleme hinweisen**
- die Token müssen auf dem Server nach auswählen und vor starten zufällig generiert werden 
    - hierzu Idee: Eine Datei (z.B. .csv) hat die Schülerklarnamen
    - dann werden zufällig in der Laufzeit dem Klarnamen ein Token zugewiesen werden
    - in ggf. einer seperaten Datei die S-Antworten speichern  in .csv => diese kann dann in der Korrektur auch iterativ ausgelesen werden 
    (2 Möglichkeiten: Eintippen, oder CSV Datei)
    - da ggf. bei der Klassenauswahl eine Schülerliste => wiederverwendetbar und automatisierbar
- ggf. prüfen, ob die Lösungen seperat von den Aufgaben gespeichert werden => hinsichtlich wie sicher die Computer in der Schule sind
- diese Antwortdatei ggf. prüfen, ob diese recht einfach verschlüsselt werden

### Server-GUI:
- im Menü-Seitenband die Einträge, die abgefrühstückt sind, entfernen und einen Zurück-Button einfügen
- "Online gehen" umbenennen in "Warteraum öffnen" => wenn dieser geöffnet wird, dann kann der L in einer L die Verbundenheit zeigen
- wenn L den Warteraum geöffnet hat, dann muss L den Test starten können 
- keine individuelle Startzeit => sondern kollektiv Zeitstart => nachträglich joinbar und in Restzeit
- hier zu eine Empfehlung angeben

### Server-Korrekturmods:
- anstelle jeden S einzeln auswählen: da nicht eine große csv mit finalem Ergebnis: sondern dort eine finale DAtei mit den aufgaben, antworten, feedback und note#
- zu dem Analyser: in Anmerkung noch mögliche Ideen für Erweiterungen schreiben (ggf. API bereitstellen, oder Aktionsobjekte)


### Frage: wie kann ein Cleint sich mit genau dem Device von dem Server in dem Einen Raum verbinden:
- auf dem Gerät der Clients eine .ini  mit der ipv4  Adresse schreiben => bei Änderungen und keiner Verbindung müsste die IPV4 ADresse vond den SUS manuell abgetippt werden
- Token länger macht, dabei implizit die ipv4 adresse abtippen, oder ggf. direkt hinein verschlüssen in das Verfahren 


### Hausaufgabe:
- Plan, welche DAten angelegt werden und benutzt werden
- Plan, wie Daten angelegt und verwaltet werden
- grobes PLanungsmodell für die DAtenobjekte erstellen
- ggf. prüfen, ob nicht SQL-Lite (ist eine kleine exe)
